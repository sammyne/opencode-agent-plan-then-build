# 需求

## 目标与背景

本次需求旨在从 [obra/superpowers](https://github.com/obra/superpowers) 项目中提取经过实战验证的 AI agent 工作流方法论，用于增强本项目的 planner 和 builder 两个 OpenCode agent。

### 背景

superpowers 是一个成熟的 AI 编码 agent 技能框架，包含 brainstorming、writing-plans、subagent-driven-development、test-driven-development、systematic-debugging 等多个 skill，形成了从需求探索到代码交付的完整工作流。其核心理念包括：

- **Hard-Gate 机制**：设计未经用户批准前禁止实现
- **TDD 铁律**：不存在没有先失败测试的生产代码
- **科学调试**：四阶段根因分析，禁止盲目修复
- **计划无占位符**：每个步骤必须包含实际内容（代码块、命令、预期输出）
- **结构化自审**：写完就审，发现问题就地修复
- **反自我合理化**：显式列出 agent 常见借口及应对

本项目的 planner 和 builder 当前已有可用的基础流程，但与 superpowers 对比存在若干可提升的维度。本次需求聚焦于**在不改变现有架构（两个独立 .md agent 文件）的前提下**，将 superpowers 中适用的方法论融入 planner.md 和 builder.md。

### 不适用的 superpowers 特性（已分析排除）

以下 superpowers 特性因架构差异或复杂度过高而不纳入本次范围：

| 特性 | 排除原因 |
|------|----------|
| Subagent-Driven Development（子 agent 分派） | OpenCode 的 agent 模式不支持 subagent 分派，且本项目是单 agent 执行模型 |
| Visual Brainstorming Companion（可视化脑暴） | 需要 Node.js 服务器 + WebSocket + 浏览器，架构复杂度远超本项目范围 |
| Git Worktree Isolation（Git 工作树隔离） | 有价值但增加大量 git 操作复杂度，建议作为独立后续需求 |
| 按任务复杂度选择模型 | 单 agent 配置无法动态切换模型 |
| Code Reviewer Subagent（代码审查子 agent） | 同样依赖 subagent 机制 |

## 功能需求列表

### 核心功能

#### A. Planner 增强

**A1. 范围检查与分解建议**
- 在阶段 1（生成需求文档）之前，增加范围评估步骤
- 如果需求涉及多个独立子系统，提示用户分解为多个独立规划，每个规划独立走 planner 流程
- 对于适当范围的需求，正常继续流程

**A2. 多方案比较**
- 在生成需求文档时，在 `## 功能需求列表` 之前增加 `## 方案比较` 章节
- 提出 2-3 个实现方案，分析各方案的优劣权衡
- 给出推荐方案及推荐理由
- 用户选择方案后，后续的功能需求和 TODO 基于选定方案生成

**A3. 需求文档自审清单**
- 在写完 requirements.md 后、提交用户确认前，执行自审清单：
  1. 占位符扫描：检查是否存在 TBD、TODO、待补充等未完成内容
  2. 内部一致性：各章节之间是否存在矛盾
  3. 范围检查：是否聚焦到可以生成单个 TODO 列表的粒度
  4. 歧义检查：是否存在可以有多种理解的需求描述，如有则明确化
- 发现问题就地修复，无需重新审查

**A4. 文件结构预规划**
- 在生成 TODO 之前，先梳理需要新增或修改的文件清单及每个文件的职责
- 作为 todo.md 中的 `## 文件结构` 章节放在任务列表之前
- 为后续任务分解提供结构依据

**A5. 禁止占位符规则**
- 在 todo.md 的任务描述中，明确禁止以下模式：
  - "TBD"、"TODO"、"待实现"、"后续补充"
  - "添加适当的错误处理" / "添加验证" / "处理边界情况"（无具体代码）
  - "为上述代码编写测试"（无实际测试代码）
  - "类似任务 N"（必须重复写出具体内容）
  - 只描述做什么、不给出怎么做的步骤（代码步骤必须有代码块）

**A6. TODO 自审清单**
- 在写完 todo.md 后、提交用户确认前，执行自审清单：
  1. 需求覆盖：逐条检查 requirements.md 中的需求是否都有对应任务
  2. 占位符扫描：检查是否存在 A5 中列出的任何占位符模式
  3. 类型/命名一致性：后面任务中用到的类型、函数名、属性名是否与前面任务中定义的一致
- 发现问题就地修复

**A7. 任务步骤细化与进度追踪**
- 每个任务内部增加可勾选的步骤（checkbox 语法 `- [ ]`）
- 步骤粒度建议对齐 "2-5 分钟单动作" 原则，但不强制每步包含完整代码块（planner 无 bash 权限无法验证代码正确性）
- 对于有测试要求的项目，步骤结构建议遵循 TDD 节奏：写失败测试 → 验证失败 → 写最小实现 → 验证通过 → 提交

#### B. Builder 增强

**B1. TDD 执行引导**
- 在执行编码任务时，如果目标项目有测试框架（从 AGENTS.md 或项目结构中识别），默认遵循 RED-GREEN-REFACTOR 流程：
  1. RED：先写一个描述期望行为的失败测试
  2. 验证 RED：运行测试，确认因预期原因失败（功能缺失，而非语法错误）
  3. GREEN：写最少量代码让测试通过
  4. 验证 GREEN：运行测试，确认所有测试通过
  5. REFACTOR：清理代码（去重、改名、提取工具函数），保持测试通过
- 如果目标项目无测试框架或 AGENTS.md 中明确不要求测试，则跳过 TDD 步骤
- 在 builder 约束中添加 TDD 铁律的常见合理化借口列表，防止 agent 跳过测试

**B2. 系统化调试流程**
- 当遇到测试失败、构建错误或意外行为时，必须遵循四阶段调试流程：
  1. 根因调查：仔细阅读错误信息、稳定复现、检查最近变更、追踪数据流
  2. 模式分析：在代码库中找到类似的正常工作的代码，比较差异
  3. 假设与验证：形成单一假设，做最小变更验证
  4. 实现修复：先写复现 bug 的失败测试，再修复根因
- 如果连续 3 次修复尝试失败，必须停下来质疑架构并向用户求助
- 明确禁止的行为：
  - "先快速修一下，之后再调查"
  - "试试改 X 看看行不行"
  - "同时改多处然后跑测试"
  - 在没有调查的情况下提出修复方案

**B3. 任务完成前自审**
- 在标记任务为 ✅ 之前，执行自审检查：
  - 完整性：是否完全实现了任务描述中的所有要求？是否遗漏了边界情况？
  - 质量：命名是否清晰准确？代码是否干净可维护？
  - 纪律：是否避免了过度构建（YAGNI）？是否只构建了被要求的部分？
- 自审未通过则继续修改，不标记完成

**B4. 任务状态精细化**
- 在 todo.md 中除了 ✅（完成）外，支持以下状态标记：
  - `⚠️`：完成但有顾虑（完成了工作但对某些方面有疑虑，需在描述中说明）
  - `🚧`：执行中（当前正在处理的任务）
  - `❌`：被阻塞（无法继续，在描述中说明阻塞原因）
- builder 遇到被阻塞的任务时，应向用户报告阻塞原因并等待指示，而非跳过或猜测

**B5. 升级协议**
- 定义明确的"何时停止并求助"条件：
  - 任务需要架构决策且存在多个合理方案
  - 需要理解任务描述之外的代码但无法找到线索
  - 对当前方案的正确性不确定
  - 任务涉及计划未预见的代码重构
  - 已经连续阅读多个文件试图理解系统但没有进展
  - 连续 3 次修复尝试失败
- 升级时必须描述：卡在什么地方、已经尝试了什么、需要什么帮助

**B6. 反合理化条款**
- 在约束部分增加常见合理化借口列表及应对：
  - "太简单不需要测试" → 简单代码也会出错，测试只需 30 秒
  - "先写完再补测试" → 后补的测试立即通过说明不了什么
  - "这次例外" → 没有例外，遵循流程
  - "手动测过了" → 手动测试不可重复、不可回归
  - "TDD 太慢了" → TDD 比调试更快
- 这些条款的作用是当 agent 试图跳过流程时触发自我纠正

**B7. 连续执行原则**
- 明确要求：执行任务时不要在任务之间暂停询问用户"要继续吗？"
- 按照 TODO 列表的依赖顺序连续执行所有任务
- 仅在以下情况停止：遇到无法解决的阻塞、歧义妨碍继续、所有任务完成
- 进度摘要和确认提示浪费用户时间，用户要求执行计划就执行计划

**B8. 分支完成工作流**
- 所有任务完成后，检测当前是否在非主分支上工作
- 如果在主分支上，跳过此流程，直接输出最终报告
- 如果在 feature 分支上，先验证测试通过，然后向用户展示 3 个选项：
  1. 本地合并回主分支并删除 feature 分支
  2. 推送到远程并创建 Pull Request
  3. 保持当前分支不动（用户自行处理）
- 执行用户选择后，再输出最终报告

## 非功能需求

- **可维护性**：每个增强点应是独立的章节/段落，便于后续单独修改或移除
- **向后兼容**：增强不能破坏现有流程的基本功能（阶段 0-4 的核心流程保持不变）
- **prompt 效率**：增加的指令文本应尽量精炼，避免不必要的冗长（总 token 敏感）
- **通用性**：增强内容不应绑定特定编程语言或框架，需要从目标项目的 AGENTS.md 中适配

## 边界与不做事项

- 不引入 subagent 分派机制（OpenCode 单 agent 模型限制）
- 不引入可视化脑暴工具（架构复杂度过高）
- 不引入 git worktree 管理（作为独立后续需求）
- 不改变 YAML frontmatter 结构（保持现有工具/权限配置）
- 不改变文件输出路径约定（requirements.md 和 todo.md 的位置不变）
- 不新增 agent 文件（只修改现有的 planner.md 和 builder.md）

## 假设与约束

- **技术假设**：产出物是两个 markdown 文件（planner.md 和 builder.md），通过修改 prompt 文本实现所有增强
- **平台约束**：在 OpenCode 平台运行，planner 无 bash 权限，builder 有完整 bash/write/edit 权限
- **token 约束**：agent 文件过长会影响 LLM 上下文利用效率，需在功能丰富和 prompt 精简之间取平衡
